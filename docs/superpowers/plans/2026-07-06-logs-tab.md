# Logs Tab Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a read-only Logs tab showing each handled URI and the final CLI command Launcher spawned.

**Architecture:** Add a Rust `logs` module for entry storage, persistence, and formatting. Thread log entry IDs through dispatch into the run and relay launch paths, then expose `list_logs` to the vanilla JS frontend.

**Tech Stack:** Rust, Tauri 2 commands/events, serde JSON, vanilla JavaScript, Vite.

### Task 2: Dispatch And Launch Integration

**Files:**
- Modify: `src-tauri/src/dispatch.rs`
- Modify: `src-tauri/src/run.rs`
- Modify: `src-tauri/src/relay.rs`

- [ ] **Step 1: Write failing integration-oriented unit tests where practical**

Add focused tests around log command formatting and store update paths from Task 1; avoid spawning external processes in unit tests.

- [ ] **Step 2: Run tests to verify failure**

Run: `cargo test logs`
Expected: tests fail until call sites use the new APIs.

- [ ] **Step 3: Thread log IDs through dispatch**

When a URI parses successfully, append a handled-URI log entry. Pass its ID into `run::launch` and `relay::start_session`. If parsing or dispatch fails, write an error log entry.

- [ ] **Step 4: Record resolved CLI calls**

In `run::launch`, update the log entry with `exec` plus `argv` before spawning. In relay, update the log entry with the editor command before spawning in both blocking and non-blocking launch paths.

- [ ] **Step 5: Run tests**

Run: `cargo test logs`
Expected: pass.
